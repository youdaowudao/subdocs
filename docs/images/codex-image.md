# 用 Codex 批量测试生图

Codex 适合把生图任务整理成可复盘的项目：先把口语需求变成清晰说明，再拆出多组提示词，生成批量脚本，记录图片结果，最后整理下一轮修改方向。

临时生成几张图片，优先使用 [无限画布](/images/infinite-canvas)。在程序里调用图片生成接口，看 [图片生成 API](/images/image-generation)。使用 ChatGPT 桌面应用中 Codex 的内置 `image_gen`，先看 [Codex 内置生图](/images/codex-image-direct)。

## 工作流程

| 阶段 | Codex 负责什么 | 产出 |
| --- | --- | --- |
| 项目整理 | 把需求整理成目标、受众、尺寸、风格和限制 | 项目说明 |
| 提示词设计 | 生成多组可测试的提示词 | 提示词清单 |
| 批量生成 | 写 API 脚本或整理画布测试方案 | 脚本 / 测试计划 |
| 结果复盘 | 根据结果提炼问题和下一轮修改方向 | 复盘记录 |
| 归档规范 | 保存提示词、图片、版本和结论 | 项目文档 |

一个生图任务单独建一个文件夹，例如：

```text
生图项目/
  需求说明.md
  提示词.md
  结果记录.md
  images/
  脚本/
```

这样做的好处是：提示词、生成结果和修改记录不会散在聊天记录里，后续复用、对比和交接都更方便。

## 整理项目需求

适合场景：需求还比较口语化，只知道“大概想要什么图”，但还没有明确尺寸、风格、用途和验收标准。

复制给 Codex：

```text
请根据当前对话和当前文件夹里的资料，整理一份可执行的图片需求说明。

要求：
1. 用 Markdown 输出。
2. 不要直接开始生图，先整理需求。
3. 当前资料不足时，只列出必须补充的问题。
4. 把需求拆成：用途、目标受众、图片数量、尺寸比例、画面主体、风格方向、色彩倾向、必须出现的元素、必须避免的元素、文字区域、交付格式。
5. 最后给出一版“可以直接拿去写提示词”的简洁总结。
```

先把“好看一点”“高级一点”“适合宣传”变成用途、尺寸、风格和验收标准，后续提示词、筛选和返工才有统一依据。

## 生成提示词方案

适合场景：已经知道要画什么，但不知道怎么写提示词，或者想一次拿到多种方向。

复制给 Codex：

```text
请根据当前文件夹里的 `需求说明.md`，生成 6 组生图提示词。

要求：
1. 每组提示词都要有明确差异，不要只是换几个形容词。
2. 每组包含：画面主体、构图、环境、光线、色彩、材质、镜头感、风格、文字区域、避免事项。
3. 提示词包含中文文字时，单独说明文字内容，避免和画面描述混在一起。
4. 输出表格，列为：方案名、适合用途、提示词、避免事项、适合尺寸。
5. 提示词要适合图片生成模型，不要写成给人类设计师的长篇散文。
6. 当前文件夹没有 `需求说明.md` 时，先根据当前对话里的需求整理一版，再生成提示词。
```

一组提示词只围绕一个明确方向，例如“产品特写”“场景海报”“社交媒体封面”分开写，后面才容易比较效果。

## 制定批量测试方案

适合场景：需要快速比较多个风格、构图或尺寸，不想一张一张临时想。

复制给 Codex：

```text
请根据当前文件夹里的 `需求说明.md` 或 `提示词.md`，整理一个批量测试方案。

要求：
1. 给出 8 个测试方向，每个方向只测试一个主要变量。
2. 变量可以是：构图、镜头距离、背景、光线、色彩、风格、材质、文案区域。
3. 每个方向给出文件名，方便保存图片。
4. 输出表格，列为：编号、测试目的、提示词、尺寸、文件名、观察重点。
5. 不要一次改变太多变量，方便判断哪一项带来了效果变化。
6. 当前文件夹没有资料时，先根据当前对话里的需求整理测试方向。
```

批量测试的重点是可比较。每张图只改一个主要变量，才能判断哪种构图、光线、风格或文案区域真正有效。

## 配合无限画布测试

适合场景：用户在无限画布里手动生成图片，但希望 Codex 先整理提示词、命名方案和迭代方向。

复制给 Codex：

```text
我要在无限画布里测试一组图片。请帮我准备画布测试材料。

要求：
1. 先读取当前文件夹里的 `需求说明.md` 或 `提示词.md`，整理 3 个主方向。
2. 每个主方向给出 3 条提示词，从保守到大胆排列。
3. 给每条提示词设计一个简短标题，方便我在画布里记录。
4. 给出图片保存命名规则，例如 01-product-closeup-a.png。
5. 最后给出一份结果记录表模板，方便我把生成结果、优点、问题、下一轮修改方向记下来。
6. 当前文件夹没有资料时，先根据当前对话里的需求整理。
```

无限画布适合多轮探索。先让 Codex 生成标题、编号、命名规则和记录表，后面就能按图片回溯提示词和修改方向。

## 生成批量脚本

适合场景：已经有项目定位，需要一次测试多条提示词、多个方向或多个尺寸。

复制给 Codex：

```text
请根据当前文件夹里的生图项目资料，写一个批量生图测试脚本。

要求：
1. 先读取当前文件夹。已有 `需求说明.md`、`提示词.md`、`结果记录.md`、图片文件或画布导出内容时，直接按这些资料做，不要让我重新粘贴。
2. 脚本用于批量测试，不要做网页、数据库、队列或复杂管理系统。
3. 根据项目定位生成 6 到 12 条测试任务，每条只测试一个主要变量，例如构图、背景、光线、色彩、风格、材质、文字区域。
4. 生成一份 `批量提示词.json`，字段包括：编号、方向、测试目的、提示词、尺寸、输出文件名。
5. 调用方式参考 `gpt-image-2` 图片生成 API：POST https://api.usegoodai.com/v1/images/generations，请求体至少包含 model、prompt、size。
6. API Key 从环境变量 `USEGOODAI_API_KEY` 读取，不要写死进脚本。
7. 不要安装新依赖；本机缺少必要工具时先说明缺什么，并等我确认。
8. 图片保存到 `输出图片/`，接口原始返回保存到 `接口返回/`，生成结果追加写入 `结果记录.md`。
9. 单张失败时记录失败原因和接口返回内容，不要让整批任务直接中断。
10. 默认顺序生成；确实要并发时最多同时 2 张。
```

批量脚本的重点不是重复接口教程，而是把“测试什么”整理清楚。每条任务只改一个主要变量，生成后才能看出哪种方向更适合当前项目。

## 复盘生成结果

适合场景：已经生成了一批图，需要筛选、总结问题，并准备下一轮提示词。

复制给 Codex：

```text
请帮我复盘这批生图结果，并整理下一轮修改方向。

请读取当前文件夹里的图片、`结果记录.md`、`批量提示词.json`、`提示词.md` 和文件名。已有记录就按记录复盘；没有记录就先按图片文件名建立编号。

1. 最接近目标的图片编号。
2. 每张图的优点。
3. 每张图的问题。
4. 问题归因：是主体、构图、光线、风格、文字区域、比例，还是提示词不清楚。
5. 下一轮应该保留什么。
6. 下一轮应该修改什么。
7. 给出 3 条优化后的提示词。
```

复盘时不要只写“好看”或“不好看”。把问题归因到主体、构图、色彩、文字区域和提示词，下一轮才知道改哪里。

## 建立项目文档

适合场景：一个生图项目会持续几轮，需要长期保存风格规范、提示词和结果。

复制给 Codex：

```text
请根据当前文件夹里的资料，为这个生图项目整理一份项目文档。

要求：
1. 读取当前文件夹里的需求说明、提示词、结果记录、图片文件名和脚本文件，不要让我重新粘贴。
2. 文档面向后续复用，不要写成聊天记录。
3. 包含：项目目标、适用场景、风格规范、提示词模板、文件命名规则、结果记录表、复盘结论、下一步计划。
4. 把已验证有效的提示词单独放到“可复用提示词”章节。
5. 把效果不好的方向放到“避免方向”章节，并说明原因。
6. 输出可以直接保存为 `项目总结.md`。
```

项目文档重点保存“有效提示词”和“无效方向”。下一次创作可以直接复用有效方向，避开已经验证失败的方向。

## 提示词基本结构

可以让 Codex 按下面结构生成或检查提示词：

| 模块 | 作用 |
| --- | --- |
| 主体 | 说明画面里最重要的对象 |
| 场景 | 说明背景、环境和使用场景 |
| 构图 | 控制近景、远景、居中、留白、海报比例 |
| 光线 | 影响质感，例如自然光、棚拍、逆光、柔光 |
| 色彩 | 控制品牌感和情绪 |
| 材质 | 强化产品、服装、建筑或道具的细节 |
| 风格 | 说明写实、插画、摄影、海报、电商图等方向 |
| 文字区域 | 需要后期加字时，提前留出干净区域 |
| 避免事项 | 说明不要出现的元素、错误文字、杂乱背景等 |

## 常见错误

| 错误 | 更好的做法 |
| --- | --- |
| 一条提示词里塞太多风格 | 每次只测试一个主要方向 |
| 只写“高级、好看、真实” | 说明主体、构图、光线、材质和用途 |
| 不记录提示词 | 每张图保存编号、提示词和结果 |
| 一次改动太多变量 | 每轮只改最影响结果的 1 到 2 个点 |
| 把 API Key 写进脚本 | 用环境变量或配置文件读取 |

Codex 的价值是把生图变成可复盘流程：需求、提示词、批量测试、结果记录和下一轮修改方向都留在项目文件夹里。
