# Codex 生图高级技巧

Codex 适合做生图前后的整理工作：把口语需求变成清晰项目说明，拆出多组提示词，生成批量脚本，记录图片结果，并把后续修改方向整理成文档。

如果只是临时生成几张图片，优先使用 [无限画布](/images/infinite-canvas)。如果要在程序里调用图片生成接口，看 [图片生成 API](/images/image-generation)。如果要使用 Codex App 内置 `image_gen`，先看 [Codex 直接生图配置说明](/images/codex-image-direct)。

## 推荐工作流

| 阶段 | Codex 负责什么 | 产出 |
| --- | --- | --- |
| 项目整理 | 把需求整理成目标、受众、尺寸、风格和限制 | 项目说明 |
| 提示词设计 | 生成多组可测试的提示词 | 提示词清单 |
| 批量生成 | 写 API 脚本或整理画布测试方案 | 脚本 / 测试计划 |
| 结果复盘 | 根据结果提炼问题和下一轮修改方向 | 复盘记录 |
| 归档规范 | 保存提示词、图片、版本和结论 | 项目文档 |

建议一个生图任务单独建一个文件夹，例如：

```text
image-project/
  brief.md
  prompts.md
  results.md
  images/
  scripts/
```

这样做的好处是：提示词、生成结果和修改记录不会散在聊天记录里，后续复用、对比和交接都更方便。

## 整理项目需求

适合场景：需求还比较口语化，只知道“大概想要什么图”，但还没有明确尺寸、风格、用途和验收标准。

复制给 Codex：

```text
请把下面这段生图需求整理成一个可执行的图片项目 brief。

要求：
1. 用 Markdown 输出。
2. 不要直接开始生图，先整理需求。
3. 如果信息不足，列出需要补充的问题。
4. 把需求拆成：用途、目标受众、图片数量、尺寸比例、画面主体、风格方向、色彩倾向、必须出现的元素、必须避免的元素、文字区域、交付格式。
5. 最后给出一版“可以直接拿去写提示词”的简洁总结。

原始需求：
【把这里换成自己的需求】
```

专业解释：

生图失败很多时候不是模型不行，而是目标不清楚。先让 Codex 整理 brief，可以把“好看一点”“高级一点”“适合宣传”这类模糊要求变成可执行约束。约束越清楚，后续生成、筛选和返工越容易。

## 生成提示词方案

适合场景：已经知道要画什么，但不知道怎么写提示词，或者想一次拿到多种方向。

复制给 Codex：

```text
请根据下面的图片项目 brief，生成 6 组生图提示词。

要求：
1. 每组提示词都要有明确差异，不要只是换几个形容词。
2. 每组包含：画面主体、构图、环境、光线、色彩、材质、镜头感、风格、文字区域、避免事项。
3. 如果提示词里包含中文文字，请单独说明文字内容，避免和画面描述混在一起。
4. 输出表格，列为：方案名、适合用途、提示词、避免事项、适合尺寸。
5. 提示词要适合图片生成模型，不要写成给人类设计师的长篇散文。

项目 brief：
【粘贴 brief 内容】
```

专业解释：

一组提示词最好围绕一个明确方向。比如“产品特写”“场景海报”“社交媒体封面”应该分开写，而不是混在同一段里。提示词中最重要的是主体、构图、光线和风格；尺寸、文字区域和避免事项用来减少返工。

## 制定批量测试方案

适合场景：需要快速比较多个风格、构图或尺寸，不想一张一张临时想。

复制给 Codex：

```text
请把下面的生图需求整理成一个批量测试方案。

要求：
1. 给出 8 个测试方向，每个方向只测试一个主要变量。
2. 变量可以是：构图、镜头距离、背景、光线、色彩、风格、材质、文案区域。
3. 每个方向给出文件名建议，方便保存图片。
4. 输出表格，列为：编号、测试目的、提示词、建议尺寸、文件名、观察重点。
5. 不要一次改变太多变量，方便判断哪一项带来了效果变化。

生图需求：
【粘贴需求或 brief】
```

专业解释：

批量测试的关键不是“多生成”，而是“可比较”。如果每张图同时改变构图、颜色、风格和主体，就很难判断哪一项有效。一次只改一个主要变量，才能逐步收敛到稳定风格。

## 配合无限画布

适合场景：用户在无限画布里手动生成图片，但希望 Codex 先整理提示词、命名方案和迭代方向。

复制给 Codex：

```text
我要在无限画布里测试一组图片。请帮我准备画布测试材料。

要求：
1. 先根据需求整理 3 个主方向。
2. 每个主方向给出 3 条提示词，从保守到大胆排列。
3. 给每条提示词设计一个简短标题，方便我在画布里记录。
4. 给出图片保存命名规则，例如 01-product-closeup-a.png。
5. 最后给出一份结果记录表模板，方便我把生成结果、优点、问题、下一轮修改方向记下来。

需求：
【粘贴需求】
```

专业解释：

无限画布适合多轮探索，但如果没有记录，很容易忘记哪条提示词生成了哪张图。让 Codex 先给标题、编号和记录表，可以把画布探索变成可复盘的设计过程。

## 生成 API 脚本

适合场景：需要批量生成、保存图片、记录提示词和文件名。

复制给 Codex：

```text
请根据 UseGoodAI 图片生成 API，帮我写一个最小可用的批量生图脚本。

要求：
1. 使用环境变量读取 API Key，不要把 Key 写死在代码里。
2. API 地址使用 https://api.usegoodai.com/v1/images/generations。
3. 模型使用 gpt-image-2，除非我另行指定。
4. 从 prompts.json 读取多条提示词。
5. 每张图保存为 PNG 文件。
6. 同时保存一份 results.json，记录文件名、提示词、模型、尺寸、生成时间。
7. 代码要简单，方便我复制后直接改。

提示词数据格式：
[
  {
    "name": "poster-01",
    "prompt": "这里写第一条提示词",
    "size": "1024x1024"
  }
]
```

专业解释：

脚本化适合重复任务，但不适合一开始就做复杂系统。先做最小脚本：读取提示词、请求接口、保存图片、记录结果。等真实重复成本出现后，再考虑队列、重试、批量管理或页面工具。

## 复盘生成结果

适合场景：已经生成了一批图，需要筛选、总结问题，并准备下一轮提示词。

复制给 Codex：

```text
请帮我复盘这批生图结果，并整理下一轮修改方向。

我会提供每张图的编号、提示词和观察结果。请按下面结构输出：

1. 最接近目标的图片编号。
2. 每张图的优点。
3. 每张图的问题。
4. 问题归因：是主体、构图、光线、风格、文字区域、比例，还是提示词不清楚。
5. 下一轮应该保留什么。
6. 下一轮应该修改什么。
7. 给出 3 条优化后的提示词。

图片记录：
【粘贴编号、提示词和观察结果】
```

专业解释：

复盘时不要只说“好看”或“不好看”。更有效的做法是把问题归因到具体维度：主体是否准确、构图是否干净、色彩是否符合品牌、文字区域是否可用。这样下一轮提示词才有明确修改方向。

## 建立项目文档

适合场景：一个生图项目会持续几轮，需要长期保存风格规范、提示词和结果。

复制给 Codex：

```text
请为这个生图项目建立一份 Markdown 项目文档。

要求：
1. 文档面向后续复用，不要写成聊天记录。
2. 包含：项目目标、适用场景、风格规范、提示词模板、文件命名规则、结果记录表、复盘结论、下一步计划。
3. 把已验证有效的提示词单独放到“可复用提示词”章节。
4. 把效果不好的方向放到“避免方向”章节，并说明原因。
5. 输出可以直接保存为 brief.md 或 results.md 的内容。

项目资料：
【粘贴 brief、提示词、结果记录和复盘结论】
```

专业解释：

生图项目最值得沉淀的是“有效提示词”和“无效方向”。有效提示词可以复用，无效方向可以避免重复踩坑。项目文档不需要复杂，但要能让下一次创作快速接上。

## 提示词基本结构

可以让 Codex 按下面结构生成或检查提示词：

| 模块 | 作用 |
| --- | --- |
| 主体 | 说明画面里最重要的对象 |
| 场景 | 说明背景、环境和使用场景 |
| 构图 | 控制近景、远景、居中、留白、海报比例 |
| 光线 | 影响质感，例如自然光、棚拍、逆光、柔光 |
| 色彩 | 控制品牌感和情绪 |
| 材质 | 强化产品、服装、建筑或道具的细节 |
| 风格 | 说明写实、插画、摄影、海报、电商图等方向 |
| 文字区域 | 需要后期加字时，提前留出干净区域 |
| 避免事项 | 说明不要出现的元素、错误文字、杂乱背景等 |

## 常见错误

| 错误 | 更好的做法 |
| --- | --- |
| 一条提示词里塞太多风格 | 每次只测试一个主要方向 |
| 只写“高级、好看、真实” | 说明主体、构图、光线、材质和用途 |
| 不记录提示词 | 每张图保存编号、提示词和结果 |
| 一次改动太多变量 | 每轮只改最影响结果的 1 到 2 个点 |
| 把 API Key 写进脚本 | 用环境变量或配置文件读取 |

一句话总结：Codex 不只是用来“写一句提示词”，更适合把生图变成可复盘的项目流程：先整理需求，再生成多组提示词，最后记录结果和下一轮修改方向。
